# 📄 APOLLO — Deep Dive from the Beginning

## The Starting Point: What Problem Exists?

A formal verification system can check whether a formal proof is correct or not almost instantaneously, but generating a completely correct formal proof with large language models remains a formidable task.

So you have this asymmetry: **verification is easy, generation is hard**. The compiler is a perfect judge — instant, free, deterministic. But getting the LLM to produce something that satisfies that judge is extremely difficult.

The community's response to this has been brute force: the usual approach is to prompt the LLM many times — up to several thousands — until one of the generated proofs passes the verification system.

Think about what this means in practice. You generate 25,600 candidate proofs, throw all of them at the compiler, keep the one that passes. The other 25,599 are discarded entirely. The compiler's detailed feedback on every single failure — the exact line, the exact tactic that broke, the exact goal state — **all of it is thrown away**.

When the formal verification system analyzes a proof, its compiler may generate a long list of issues including syntax errors, incorrect tactics, open goals, and more. In the current approach, all of that information from the compiler remains unused, as the LLM's proof generation is not directly engaged with the compiler errors.

That's the core waste APOLLO is targeting. Not "generate better proofs from scratch" — but "stop throwing away the repair signal that already exists."

---

## The Key Observation That Motivates Everything

Often the LLM is capable of producing a valid proof sketch — the high-level mathematical strategy is correct — but it often struggles with closing fine-grained goals. Statements like h4, h5, h7, h8 may be structurally correct but the generated proofs for each sub-lemma throw compilation errors.

This is a really important distinction. The failure mode isn't usually "the LLM has completely wrong mathematical understanding." It's more surgical: **the overall proof structure is right, but specific leaves of the proof tree fail to compile**.

A human mathematician collaborating with Lean would:
1. See the proof mostly works
2. Find the specific failing step
3. Focus their attention entirely on fixing that one piece
4. Leave the rest intact

APOLLO wants to automate exactly that workflow.

## The Five-Stage Pipeline — What Each Stage Is Actually Doing

### Stage 1 — Syntax Refiner

This is the simplest stage and it targets a specific failure class: **the LLM doesn't know Lean4 syntax**.

The Syntax Refiner catches and corrects superficial compilation errors: missing commas, wrong keywords such as Lean3's `from` vs Lean4's `:= by`, misplaced brackets. These syntax errors arise when a general-purpose LLM has not been specialized on Lean4 syntax.

Notice what this implies: general-purpose models like o3-mini and o4-mini have strong mathematical reasoning but default to Lean3 syntax or introduce trivial formatting mistakes. Their proof *logic* can be completely correct while the proof *text* fails to compile for purely syntactic reasons. The Syntax Refiner peels off that surface layer so the real mathematical content can be evaluated.

This module is primarily applicable to general-purpose models not explicitly trained for theorem proving in Lean. The Syntax Refiner usually does not get invoked for proofs generated by LLMs trained for formal theorem proving.

So this stage is almost exclusively doing work for models like o3-mini, not for specialized provers like Goedel.

---

### Stage 2 — The Sorrifier (the most important stage)

This is the heart of APOLLO and it needs careful unpacking.

After the Syntax Refiner, you have a proof that's syntactically valid but still fails mathematically. The compiler points to specific failing blocks. The Sorrifier's job is to **localize failures** without breaking everything else.

The Sorrifier patches any remaining compilation failures by inserting Lean's `sorry` placeholder. It first parses the failed proof into a tree of nested proof-blocks, then compiles the proof with Lean REPL, detects the offending line or block, and applies one of three repairs: line removal if a single tactic is invalid but its surrounding block may still succeed; block removal if the entire sub-proof is malformed; or inserting `sorry` if the block compiles but leaves unsolved goals open.

`sorry` in Lean is a special keyword that says "I claim this is true, trust me for now." The compiler accepts it and moves on. So the Sorrifier is essentially doing **surgical fault isolation** — it replaces every failing block with a `sorry` placeholder until the entire proof compiles cleanly, with `sorry`s marking exactly where work remains.

This procedure repeats until the file compiles without errors. At that point every remaining `sorry` marks a sub-lemma to be proved in later stages. Each `sorry` block corresponds to a sub-problem that the LLM failed to prove.

The output of the Sorrifier is therefore a **map of what's broken**. Not just "proof failed" but "these exact three sub-lemmas need to be proven, and here is their full local context — all the hypotheses, definitions, and previously proven lemmas available at that point."

This is the key transformation: from one big failed proof → a collection of small, well-defined, isolated sub-problems.

---

### Stage 3 — Auto Solver

Now you have a list of `sorry`-marked sub-goals. Before invoking the LLM (which is expensive), try to close them with built-in Lean tactics.

The Auto Solver targets each `sorry` block in turn. It first invokes Lean4's `hint` to close the goal. If goals persist, it applies built-in solvers like `nlinarith`, `ring`, `simp` wrapped in `try` to test combinations automatically. Blocks whose goals remain open stay marked with `sorry`.

`nlinarith` handles nonlinear arithmetic. `ring` handles ring identities. `simp` handles simplification. These are powerful automated tactics that can close a surprising number of sub-goals entirely mechanically — no LLM needed, zero tokens spent.

The insight here is that many of the "failing sub-lemmas" are actually trivial algebraic facts that the LLM wrote clumsy proofs for. The Auto Solver just sweeps those up cleanly and cheaply, leaving only the genuinely hard sub-goals for the LLM.

---

### Stage 4 — Recursive LLM Repair

Now you're left with sub-goals that neither the Sorrifier heuristics nor the Auto Solver could close. These are handed to the LLM — but crucially, **not the whole theorem**.

Apollo considers each remaining `sorry` as a new lemma — it extracts its local context including hypotheses, definitions, and any lemmas proved so far — and recursively tries to prove the lemmas by prompting the LLM and repeating the whole process of verification, syntax refining, sorrifying, and applying the Auto Solver.

This is the recursive structure that makes APOLLO powerful. Each sub-lemma gets the full pipeline applied to it. If that sub-lemma itself fails, it gets Sorrified into sub-sub-lemmas, which get Auto-Solved or sent back to the LLM. The recursion continues up to a user-specified depth `r`.

The LLM at each recursive call is working on a **tiny, focused, well-specified problem** — not the whole theorem. A theorem might need 100 lines of proof. The sub-lemma the LLM is asked to fix might need 1-5 lines. This dramatically increases the probability the LLM succeeds.

---

### Stage 5 — Proof Assembler

Finally, the Proof Assembler splices all repaired blocks back into a single file and verifies that no `sorry` or `admit` commands remain. If the proof still fails, the entire pipeline repeats up to the user-specified recursion depth `r`, allowing further rounds of refinement.

The Assembler is essentially a stitching and final verification step. It reassembles the tree of sub-proofs into one complete proof and runs it through Lean one final time to confirm there are no remaining holes.

## The Full Pipeline as a Loop

```
Initial LLM proof attempt
         ↓
┌─────────────────────────────────────────────────┐
│  1. SYNTAX REFINER                               │
│     Fix surface syntax errors (Lean3 vs Lean4)  │
│         ↓                                        │
│  2. SORRIFIER                                    │
│     Parse proof tree → find failing blocks      │
│     Replace each failure with sorry              │
│     Output: { sorry₁, sorry₂, ... } = sub-goals │
│         ↓                                        │
│  3. AUTO SOLVER                                  │
│     Try nlinarith / ring / simp on each sorry   │
│     Close trivial sub-goals for free            │
│     Remaining: hard sub-goals only              │
│         ↓                                        │
│  4. RECURSIVE LLM REPAIR                         │
│     For each remaining sorry:                    │
│       Extract local context                      │
│       Prompt LLM on this one sub-goal only      │
│       Apply stages 1-4 recursively (depth r)    │
│         ↓                                        │
│  5. PROOF ASSEMBLER                              │
│     Stitch sub-proofs back together             │
│     Final Lean compile: any sorry left?         │
│     If yes → repeat whole loop                  │
│     If no → DONE ✓                              │
└─────────────────────────────────────────────────┘
```

## What This Achieves

The fundamental shift is from **sampling-based search** to **repair-based search**.

Old approach: generate K full proofs, hope one compiles. Cost scales linearly with K.

APOLLO: generate one proof, repair it surgically, each repair attempt is tiny and targeted. Goedel-Prover-SFT's accuracy jumps from 64.7% to 65.6% while the sampling budget drops by two orders of magnitude — from 25,600 generated samples to only a few hundred on average.

And for general-purpose models the effect is even more dramatic: o3-mini and o4-mini jump from 3-7% accuracy to over 40% accuracy, demonstrating that in many scenarios discarding and regenerating the whole proof is overly wasteful — with a little help from the Lean compiler, better accuracies are achievable while sampling far fewer proofs.
